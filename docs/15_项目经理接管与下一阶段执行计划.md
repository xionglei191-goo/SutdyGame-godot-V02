# 项目经理接管与下一阶段执行计划

> 更新日期：2026-06-04

## 1. 项目目标与首期成功标准

StudyGame V02 是面向约 10 岁、兼容 0 基础儿童的移动端英语生活冒险游戏。产品以 Sunshine Town、A-Z 记忆宫殿和宠物生活为核心，通过生活事件、收集、小游戏和英语环境实现低压力学习。

第一阶段只验证一个完整闭环：

```text
Home 遇见宠物
  -> 宠物饿了
  -> Letter Snake 获得 coins 和卡片进度
  -> Supermarket 购买食物
  -> 喂宠物
  -> 宠物反馈、Memory Card 和本地记录更新
```

首期成功标准：

- Godot 4.x 工程可在桌面开发环境稳定运行。
- 运行时从 JSON 加载地图和核心内容，不依赖 EditorOnly 节点。
- 上述主循环可从头到尾跑通。
- 孩子端无考试、课程表、失败惩罚或家长报告。
- 地图、A-Z、卡片状态和奖励链有自动化检查。

## 3. 下一里程碑：可运行技术薄片

里程碑目标不是完成完整地图编辑器或全部系统，而是建立可持续扩展的最小技术薄片。

### P0 开工任务

| 顺序 | Task ID | 交付物 | 完成门槛 |
|---|---|---|---|
| 1 | V02-CORE-001/002 | Godot 工程与目录骨架 | 编辑器可打开，主场景可运行 |
| 2 | V02-MAP-001 | 核心 Resource 与 JSON schema 示例 | World/Place/District/Road 可加载 |
| 3 | V02-CORE-004 | Local SaveService stub | profile 和 GameState 可往返保存 |
| 4 | V02-MAP-008 | 第一批数据校验测试 | 非法 ID、区域和 A-Z 顺序可拦截 |
| 5 | V02-RUNTIME-001/002 | 最小 RuntimeMapBuilder | 从 JSON 显示地面、道路和地点占位 |
| 6 | V02-LOOP-001 | 宠物食物循环状态与奖励规则 | 事件输入、状态变化、奖励可实现 |

### P1 紧随任务

- 实现 `MemoryCardService`、`PetService`、`ShopService` 和 `QuestEventService` 最小版。
- 制作 Home、Supermarket 和 Letter Snake 占位场景。
- 接通 coins、购买、喂食、卡片进度和本地保存。
- 增加主循环 smoke test，并进行手机/平板触控检查。

阶段 2 的完整地图编辑器、完整 26 个 anchor 互动、真实语音、真实 AI 和家长后台 UI 均不进入本里程碑。

## 4. 开工前决策

| 决策项 | 建议 | 截止点 |
|---|---|---|
| 首批启用 anchor 数量 | 固定为文档实际列出的 9 个：A/B/C/D/K/O/S/T/W | 创建首批 A-Z JSON 前 |
| 首个可玩地图范围 | 仅 Home、Town 起步区、Supermarket；其他地点先放占位 marker | RuntimeMapBuilder 开发前 |
| 首个宠物形态 | 使用 Sunny/Dog 占位，避免先做宠物选择系统 | 主循环实现前 |
| 自动测试方案 | 使用仓库内自建 headless runner，避免首期引入测试插件依赖 | 已确认 |
| 移动端目标平台 | Android 横屏优先；1280x720 逻辑分辨率，最低验证视口 960x540；iOS 保持架构兼容 | 已确认 |

## 5. 风险与控制

| 风险 | 影响 | 控制措施 |
|---|---|---|
| 同时建设过多系统 | 主循环迟迟不可玩 | 以技术薄片为唯一 P0，接口只实现闭环所需字段 |
| 地图编辑器过早复杂化 | 工具投入挤压玩法验证 | 先用手工 JSON + 校验跑通运行时，再扩展可视化编辑 |
| 内容与数据 schema 并行漂移 | 返工和加载失败 | schema 示例先行，内容 Agent 只按已验收字段交付 |
| 占位资产被误当成最终风格 | 后续视觉返工 | AssetResolver 管理占位资源，禁止玩法脚本硬编码路径 |
| 儿童端出现学习压力表达 | 偏离产品定位 | QA 建立禁用词与孩子端可见性检查 |
| 真实 AI/语音提前接入 | 扩大隐私和技术风险 | 第一阶段严格使用 stub，不申请录音或联网权限 |

## 6. 执行与汇报机制

- 每个任务使用 `docs/collaboration/任务交接模板.md` 分配，并明确 Owner、范围和验收标准。
- 开发任务必须同时提交运行方式、验证结果、风险和待确认项。
- 每次里程碑评审按“可运行闭环、数据合同、儿童体验、移动端表现”四项验收。
- 新需求默认进入待办，不得绕过当前 P0 直接扩大范围。
- 项目经理维护任务状态、依赖、决策记录和风险台账；技术负责人维护架构与数据合同。
