### 接管判断

StudyGame V02 已经具备可移动小镇、NPC、资源、商店、小屋、相册、A-Z anchor、本地内容合同、5 分钟可玩纵切、7 天回访节奏、Home / School P0 路线、scene-native 架构、Map Editor dogfood、首批 story / motion / UI feedback 资产和第二批 story prop production 资产。问题不在“系统没有做”，也不只是几处视觉 bug；问题是当前生产模型仍是 `world_map` 数据、cell、place、anchor、NPC 和功能入口先行，再把视觉层贴到运行时上。这个模型会自然生成“系统索引式地图”，而不是 artpass003 所要求的 home-centered cozy town gameplay frame。

Round119 的 `approved` 从 V02.38 起改解释为旧阶段 `functional_pass`：证明入口、基本可读性和旧 smoke 成立，但不代表 `final_approved`。Round159 的 `visual_candidate` 在二次视觉复核后也只保留为 `visual_scaffold`：证明 no-full-map 和 modular layer 工程方向可行，但没有达到目标画面。`final_approved` 必须在 V02.39 通过 target frame、runtime visual match、真实入口和 PM / Art Direction 复核重新判定。

系统性根因拆分如下：

- Art：缺少可直接指导 Godot 落地的 target frame / art bible 合同，只有方向参考和后验截图 gate。
- Architecture：逻辑地图和视觉布局耦合，导致 26 anchors、place labels、NPC、resource 和 HUD 都试图同时证明自己存在。
- Code：fallback / context marker / low-alpha old scene 残留对功能友好，但对最终视觉 proof 有害。
- Planning：首屏承载过多系统展示，内容批量和批准线推进早于目标画面锁定。
- UX/UI：HUD 和标签仍承担解释系统的职责，而不是安静支持小镇生活。
- QA：headless、layer count、入口 smoke 和 no-full-map gate 被误当作视觉批准的充分条件。
- Tooling：Map Editor 适合 authoring 逻辑地图，不适合单独完成 artpass 级别的首屏视觉构图。

## 历史记录：旧阶段 1 技术薄片计划

## 1. 项目目标与首期成功标准

StudyGame V02 是面向约 10 岁、兼容 0 基础儿童的移动端英语生活冒险游戏。产品以 Sunshine Town、A-Z 记忆宫殿和宠物生活为核心，通过生活事件、收集、小游戏和英语环境实现低压力学习。

第一阶段只验证一个完整闭环：

```text
Home 遇见宠物
  -> 宠物饿了
  -> Letter Snake 获得 coins 和卡片进度
  -> Supermarket 购买食物
  -> 喂宠物
  -> 宠物反馈、Memory Card 和本地记录更新
```

首期成功标准：

- Godot 4.x 工程可在桌面开发环境稳定运行。
- 运行时从 JSON 加载地图和核心内容，不依赖 EditorOnly 节点。
- 上述主循环可从头到尾跑通。
- 孩子端无考试、课程表、失败惩罚或家长报告。
- 地图、A-Z、卡片状态和奖励链有自动化检查。

## 5. 风险与控制

| 风险 | 影响 | 控制措施 |
|---|---|---|
| 同时建设过多系统 | 主循环迟迟不可玩 | 以技术薄片为唯一 P0，接口只实现闭环所需字段 |
| 地图编辑器过早复杂化 | 工具投入挤压玩法验证 | 先用手工 JSON + 校验跑通运行时，再扩展可视化编辑 |
| 内容与数据 schema 并行漂移 | 返工和加载失败 | schema 示例先行，内容 Agent 只按已验收字段交付 |
| 占位资产被误当成最终风格 | 后续视觉返工 | AssetResolver 管理占位资源，禁止玩法脚本硬编码路径 |
| 儿童端出现学习压力表达 | 偏离产品定位 | QA 建立禁用词与孩子端可见性检查 |
| 真实 AI/语音提前接入 | 扩大隐私和技术风险 | 第一阶段严格使用 stub，不申请录音或联网权限 |

## 6. 执行与汇报机制

- 每个任务使用 `docs/collaboration/任务交接模板.md` 分配，并明确 Owner、范围和验收标准。
- 开发任务必须同时提交运行方式、验证结果、风险和待确认项。
- 每次里程碑评审按“可运行闭环、数据合同、儿童体验、移动端表现”四项验收。
- 新需求默认进入待办，不得绕过当前 P0 直接扩大范围。
- 项目经理维护任务状态、依赖、决策记录和风险台账；技术负责人维护架构与数据合同。
