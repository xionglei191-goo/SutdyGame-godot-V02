- place、anchor、NPC、resource 都要有 visual proxy，而不是直接由数据对象生成最终图像。
- 使用 Godot 的 Node2D/YSort/CanvasLayer 分层：ground、path、region、building、prop、actor、feedback、HUD 明确分层。

推荐路线不是立刻转 3D。对当前项目来说，最稳的是 **2D 预渲染 / 手绘 cutout + 2.5D 层级 + Y-sort**。真 3D 会带来相机、模型、动画、灯光、性能和资产管线重写，现阶段风险太大。

**3. 代码维度**

本质问题：现在代码在“兜底画出来”，而不是“严格按视觉合同渲染”。

很多 fallback、placeholder、alpha 降噪、程序化 sprite 在功能阶段很有用，但 final visual 阶段会害人。它们让系统永远能画出点东西，也就永远能把不该出现的东西带进 proof。

要解决：

- production visual mode 下，缺资产应该 fail gate，不应该自动画 placeholder。
- `main.gd` 不应该继续维护大量 `_logical_asset_texture_keys` 和颜色 fallback；这些应该进入 ThemeProfile / VisualAssetSpec。
- 每个视觉 asset spec 至少包含：path、category、pivot、default_size、scale_range、z_policy、shadow_policy、allowed_runtime_modes。
- 禁止大场景 `place_assets` 进入 child runtime visual layer，除非明确是 `reference_only` 不渲染。
- Anchor badge、place label、hotspot glow 都要变成状态机：默认、靠近、聚焦、交互中，而不是常驻。

**4. 策划维度**

本质问题：P0 首屏承载了太多策划结构。

A-Z 记忆宫殿、Home、School、Shop、Animal Park、Coast、NPC、资源、背包、相册、家具系统都想在首屏证明自己存在。结果画面不是小镇，而是“系统索引”。

要解决：

- P0 首屏只服务一个核心体验：孩子在 Home 附近的小镇日常中走一走、看看、遇见一两个生活物件。
- A-Z 是世界结构，不是首屏全量 UI。26 anchor 可以在逻辑上常驻，但视觉上必须分区域、分距离、分状态出现。
- Home / School / Shop / Animal Park / Coast 不必同屏都强表达。artpass003 的方向图能同屏，是因为它是美术构图，不是所有系统入口都同等权重。
- 每轮新增内容必须先问：它会不会让首屏更像小镇？如果只是证明系统存在，就不应该进 visual gate。

**5. UX / UI 维度**

本质问题：HUD 还在解释系统，而不是陪伴场景。

目标是 Apple-like translucent glass，但不是“白色透明矩形”。真正的 UI 应该轻、短、图标优先、少解释，把主视觉还给小镇。

要解决：

- 顶部 HUD 拆成小型状态组，不横跨整屏。
- 底部 dock 图标优先，短标签辅助；不要像表单按钮。
- Prompt 只在上下文出现，不常驻讲规则。
- 设置、背包、商店、Home 编辑都应是临时状态；非交互状态下小镇画面必须独立成立。
- UI proof 要看“关闭面板后的世界是否仍然好看”，不能只验面板可读。

**6. QA / PM 维度**

本质问题：自动化测试被用来替代艺术判断。

测试可以拦截错误：旧底图、节点重复、遮挡、尺寸、禁用文案。但测试不能判断“像不像目标小镇”。这必须有 Art Direction gate。

要解决：

- 状态语义要再细化：
  - `functional_pass`：能玩。
  - `visual_scaffold`：视觉架构存在。
  - `art_target_locked`：目标静帧被认可。
  - `runtime_visual_match`：Godot 同屏接近目标静帧。
  - `final_approved`：人工 PM / Art Direction 批准。
- 每个 visual gate 都要提交 side-by-side：artpass003 / target paintover / Godot runtime screenshot。
- QA 自动化只做底线；最终视觉批准不能由 count、alpha、截图尺寸推导。

**7. 工具链维度**

本质问题：Map Editor 是逻辑生产工具，不是视觉构图工具。

当前 Map Editor 很适合维护 cell、place、resource、routine，但它无法产出 artpass003 那种画面。视觉构图需要另一套 authoring 数据。

要解决：

- 保留 Map Editor 管逻辑。
- 新增 Visual Layout authoring：可以是 JSON、Godot scene、或后续编辑器面板。
- 视觉布局记录 screen position / layer / scale / focus group / asset variant。
- 逻辑 place_id 和 visual node 通过 stable ID 绑定，而不是共用同一套坐标。

**我会怎样制定真正可达的路线**

第一阶段：目标翻译  
产出一张 `1280x720` 首屏目标图和一份 Godot 风格合同。明确：视角、比例、颜色、HUD 位置、首屏只显示哪些 anchor、哪些东西不显示。

第二阶段：视觉架构拆分  
新增 `VisualLayout` 数据和 `WorldVisualComposer`，把逻辑地图与视觉表现分开。旧 `TownStage` 的“全量画数据”降级为 debug / editor 用途。

第三阶段：统一资产包  
按目标图生产同源资产。不是补几张 PNG，而是首屏环境 kit：地面、道路、建筑、树、围栏、水岸、生活 props、角色、UI。

第四阶段：Godot runtime 复现目标静帧  
先不追完整交互，只追一张 Godot 截图接近目标图。静帧不成立，交互越多越乱。

第五阶段：把交互接回去  
接回 `看看`、NPC、Home、Shop、A-Z album，但全部通过 visual proxy 和 focus state，不再全量裸露。

第六阶段：视觉 gate  
同轮导出 target / runtime / annotated diff。人工通过才叫 `runtime_visual_match` 或 `final_approved`。

所以我的真实判断是：**Godot 可以做到目标方向，但当前项目不能靠修补做到。**  
要达到目标，需要把 V02.38 后续从“视觉纠偏”升级为“Godot 视觉生产体系重建”：美术先定义画面，架构承接画面，策划收缩首屏，代码严格渲染合同，QA 只守底线，Art Direction 做最终判断。
