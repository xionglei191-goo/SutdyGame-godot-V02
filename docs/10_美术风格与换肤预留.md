# 美术风格与换肤预留

> 当前定位：V02 美术生产基线。  
> 本文不讨论多主题并行落地，只规定当前 storybook / cozy town 美术生产、命名、逻辑 asset ID 和验收方式。

## 初期风格

V02 初期只采用一种美术风格，避免范围失控。

目标用户是 10 岁左右、四年级小女孩。风格关键词：

- 可爱。
- 温暖。
- 明亮。
- 梦幻但不幼稚。
- 小镇生活。
- 宠物陪伴。
- 收集感。
- 轻 2.5D。
- 绘本感。

## 美术生产分类

首批素材按以下分类生产和跟踪：

- 角色 / NPC。
- Sunny / 宠物。
- 小镇场景。
- A-Z 锚点物件。
- 家具与家园。
- UI 与图标。
- 音效与环境氛围。

每个素材都要记录：

- 用途。
- 逻辑 asset ID。
- 当前状态：占位、半正式、正式。
- 建议尺寸或比例。
- 触控 / 观看场景。
- 替换目标。
- 验收截图要求。

## 统一范围

同一阶段内需要统一：

- 地图。
- 角色。
- 宠物。
- NPC。
- UI。
- 图标。
- 记忆卡。
- 小游戏。
- 音频情绪。

不同时尝试像素风、3D Q 版、扁平卡通、水彩等多套方向。

## AssetResolver 约束

资产系统不能把具体路径写死在玩法逻辑里。玩法脚本只能通过逻辑 ID 请求资源。

建议设计：

```text
ThemeProfile
AssetResolver
```

运行时通过接口取资源：

```text
get_npc_sprite("mina")
get_pet_sprite("sunny")
get_tile_atlas("world")
get_card_art("a_apple_core")
get_ui_skin("primary_button")
```

## 命名与交付

- 使用稳定逻辑 ID，不直接依赖文件名作为玩法约定。
- 角色、场景、家具、UI 图标和音效都要有可追踪的替换状态。
- 首批素材优先替换首屏、核心交互和高频 UI。
- 验收时必须给出截图或录屏证据。
