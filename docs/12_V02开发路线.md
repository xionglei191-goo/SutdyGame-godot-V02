## 四条并行路线

### 工程路线

- 维护现有可玩闭环，不把已完成系统改回调试面板或隐藏 contract 按钮。
- 所有“可玩功能”必须有孩子端可见入口，且至少一个测试通过可见按钮或面板按钮触发。
- 玩法脚本不得硬编码具体素材路径，只能通过逻辑 asset ID 和 `AssetResolver` 请求资源。
- 新增数据内容优先改 JSON / 数据合同 / focused test，避免把内容规则重新塞回 `scripts/main.gd`。
- V02.12 的世界规划先进入独立 `az_world_plan` 数据合同；未通过规划合同前，不把 26 个 anchor 一口气写成运行时主线任务。
- V02.22 以后，地图采用“authoring scene + runtime JSON + scene 节点表现”的架构：`TownMapAuthoring` 可视编辑 marker 并导出 dictionary，运行时仍以 `world_map.json` 为事实来源，渲染、输入、角色、NPC、资源、anchor、可摆物件和 UI 已拆出 `TownStage`、`PlayerActor`、`NPCActor`、`InteractableObject`、`HomeRoom`、`TownHUD`、Backpack / Shop / Settings / Album panel scenes。
- 拆分 scenes 必须保持向后兼容：每拆出一个 scene，先保留原玩家路径、save key、logical asset ID、A-Z anchor ID / route_order 和 focused/headless 验证，再删除旧生成逻辑。
- V02.23 先做 PM 批准线，不直接跳运行时实现；`tests/test_v0223_ui_panel_scene_split.gd` 属于 Round113 HIDDENGRID-007 回归证据，不得被误用为 V02.23 product approval 证据。
- V02.25 写回工具必须双阶段提交：authoring 层生成候选 dictionary，sync service 校验后合法才写；合同失败、26 A-Z 退化或 JSON 无法加载时不得覆盖 `data/maps/world_map.json`。
- V02.27 Map Editor 继续是场景内工具，不暴露给孩子端 runtime；Place、road、collision、interaction、resource、NPC routine 和 A-Z anchor 只能先进入候选 state，保存前必须走对应分文件验证和安全写入。
- V02.28 dogfood 必须优先使用场景内 Map Editor 完成候选修改和保存验证；允许为测试和文档读取 JSON，但不得绕过 `MapEditorSyncService` 直接手改正式数据来冒充工具生产。

### 策划路线

- NPC 问候、关系阶段、轻委托、商店轮换、节日天气和 A-Z 场景故事要形成可复用生产规范。
- 每条每日委托必须有 NPC、目标、奖励、关系反馈、重复规则、儿童安全文案和可见操作入口。
- 经济设计保持非惩罚：买不起只给温和反馈，不制造打卡压力、损失惩罚或强消耗循环。
- 英语继续作为环境层、收藏层、NPC 短句和记忆宫殿故事层，不作为考试、课程表或主线门槛。
- 后续课本主线从 Home / School 的 0 基础桥接开始，知识点以场景物件、NPC 短句、相册故事和轻委托挂接到 anchor，而不是以课程地图呈现。
- V02.14 P0 只选 Home、Home-School Walk、School Gate、School Yard、Shop 和 Sunny 角落；Bookshop、Bus Stop、Park、Theatre、Music Corner、Far Edge 继续作为 P1/P2，不阻断第一条可玩主线。
- V02.15 继续只强化 Home / Walk / School Gate / School Yard / Home Return，不新增课程 UI；School 是生活地点、问候地点和操场回访地点，不是课堂或考试空间。
- V02.16 只整理现有 P0/P1 体验，不新增课程 UI、不扩 Bookshop / Bus Stop 为 P0、不改变 A-Z anchor 结构；所有文本审校以温暖、生活化、短句为准。
- V02.17 只把既有 `az_world_plan` 运行时落地，不重新命名字母、不改变 `anchor_id` / `core_word` / `route_order`，也不要求孩子按 A-Z 顺序走完整路线。
- V02.18 只做地图视觉可读性、导航和探索反馈抛光，不新增课本主线、不扩远郊 P0、不把字母徽章做成主任务或完成率。
- V02.19 只做实际地图和资产生产替换并收口为 1280 art baseline，不新增玩法系统、不改变 26 anchor 稳定 ID / 坐标语义 / route_order，不把 School 做成课堂或考试空间，也不标 product complete。
- V02.20 只做可居住小镇 playgate 审计、返修规划、旧路径回归和 approved 判定，不新增课程 UI、不扩远郊 P0、不把 Letter Snake 重新变成主线；已收口为 V02.21 输入，不标 product complete。
- V02.21 只做 V02.20 已审计出的居住感返修实现，不新增内容扩展、不做 960x540 专项、不改变 A-Z ID / route_order。
- V02.22 只扩“生活可编辑性”和“世界 routine”，不扩课程内容：户外装饰、资源刷新、NPC 日常移动、地点到达反馈和 A-Z 生活物件回访都必须像小镇生活，不写成任务表、学习路线、字母打卡或每日必须事项。
- V02.23-V02.26 继续只服务生活小镇批准、居住感、工具安全和小批量内容，不新增课程 UI、远郊 P0、字母顺序任务或 product-complete 声明。
- V02.27 不改变世界叙事内容，只提升生产工具；A-Z anchor 位置迁移只能在专门模式中进行，稳定 ID、letter、core_word、route_order、card_id 和 audio_id 永远不可编辑。

### 美术路线

- 初期只采用单一 Animal Crossing-like cozy town 风格，不并行扩展多套主题；明确不采用绘本 / storybook / picture-book 叙事插画质感。
- HUD、弹层、按钮底板和高频 UI 容器采用 Apple-like translucent glass 方向：轻透明、柔和模糊、边界清楚、短文本可读、触控态稳定。
- 所有素材需要用途、逻辑 asset ID、尺寸/比例、状态、替换优先级和验收截图。
- 首批素材范围覆盖角色/NPC、Sunny/宠物、小镇场景、A-Z 锚点物件、家具与家园、UI 与图标、音效与环境氛围。
- UI 和 HUD 逐步从纯文字表达转向图标+短文本表达，但必须保证移动端可读和可触控。
- `production` 只表示资源可进入游戏集成，不等同于美术最终质量；历史阶段曾要求 `approved` 具备 PM / Art Direction 确认和 1280x720、960x540 截图证据。自 2026-06-06 起，后续未完成任务的 `approved` 阻塞性截图证据只要求 1280x720，960x540 进入全量开发完成后的版本适配专项。
- V02.7A 前不得继续把缺少截图证据的生成图或程序占位图当作产品视觉完成。
- V02.16 截图复核必须区分 `production` 和 `approved`：可集成不等于可批准，缺少 1280x720 与 960x540 证据的视图不得标为 `approved`。
- V02.17 地图资产必须按 logical asset ID 接入；每个 anchor 在 960x540 下至少能读出主轮廓，不能依赖小字或裸字母牌证明存在。
- V02.18 的地图抛光必须继续以生活物件为主、字母徽章为辅助；任何新视觉层都要在双视口截图中证明无遮挡、可读且不压住 HUD / 交互按钮。
- V02.19 的美术生产按 P0 可玩视野优先：先做视觉方向确认包，再替换世界地图底图 / 区域块、Home / Home-School Walk / School Gate / School Yard、Town Plaza / Shop / Bookshop / Bus Stop，再跟进 26 anchor 物件、角色 / 宠物和 UI 图标；后续所有截图批准先看 1280x720，960x540 不删除、等全部开发完成后统一适配。
- V02.19 所有新资产必须有 stable logical asset ID、建议尺寸、用途、状态、替换目标和截图验收点；runtime 不得硬编码具体图片路径。
- V02.20 的美术与 UX 判断以“能生活的小镇”优先于“素材已接入”：审计居民存在感、可停留空间、资源 / 家具用途感、anchor 自然嵌入、HUD / 弹层遮挡和 `production` / `approved` 区分。
- V02.21 的美术与 UX 实现优先修复 Home needs_fix、School line proof 缺口、Shop 空间感、anchor / resource / NPC 热点互吞和 HUD / 弹层可触可读问题。
- V02.22 的视觉原则是“格子隐藏，空间连续”：允许底层使用 60x34 或后续更细的 cell 管理占位，但孩子端不显示调试网格、瓦块边界、坐标或编辑器符号；户外装饰和 A-Z anchor 先作为生活物件融入底图和 Sprite 层，字母徽章继续只作辅助识别。
- V02.23 视觉批准只看当前同轮 1280 proof 和 PM / Art Direction 判定；Album 之外的六个 `needs_fix` 视图不得因为 V02.22 scene-native 完成、资产为 `production` 或旧 proof 存在而自动升级。

- 玩家路径 smoke 继续覆盖小镇、背包、商店、小屋、相册、anchor、NPC、资源和家具操作。
- 后续所有未完成任务的截图验收以 1280x720 作为阻塞门槛，检查遮挡、文字溢出、安全区、按钮尺寸和关闭路径；较小横屏视口如 960x540 保留到全部开发完成后的版本适配专项，不阻塞中间阶段任务。

## 持续门槛

- 每个可玩功能必须有玩家可见入口，隐藏 contract 按钮不能作为完成依据。
- 每个素材必须有稳定逻辑 asset ID，不能由玩法脚本硬编码具体路径。
- 每个新词必须绑定 `letter`、`core_anchor_id`、`world_place_id`、`story_memory`、`visual_hook` 和 `review_path`。
- 每次 UI 或美术大改必须补截图验收，不能只跑 headless。
- 儿童端不得出现失败惩罚、课程压力、家长报告口吻、开放陌生人社交或工程调试文案。
- 隐藏网格不得在孩子端变成可见瓦块、坐标、调试路径或编辑器术语；scene 组件化不得破坏已完成的可见入口、保存结构和 A-Z 记忆宫殿稳定编码。
