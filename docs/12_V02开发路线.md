## 当前纠偏事实

- 当前唯一阶段改为 `V02.39 Godot 视觉生产体系重建`。V02.38 只证明工程上可以摆脱 full-map background，并形成 terrain / chunk / prefab / prop / actor / glass UI 的脚手架；它没有证明首屏达到 `artpass003_visual_direction`。
- 根因不是某几处贴图、透明度或 HUD 小问题，而是生产模型错误：项目仍以 `world_map` 数据、cell、place、anchor、NPC 和功能入口为第一驱动，再把视觉资产贴到运行时上；目标方向需要先锁定 1280 gameplay target frame，再把视觉构图、层级、透视、密度和交互代理绑定回 Godot。
- 当前画面的主要结构性问题包括：首屏缺少 home-centered cozy town 构图；旧 scene / place context 仍以低透明残留污染画面；26 个 A-Z anchor 和 badge 仍像系统索引；地点标签偏 debug / UI signage；资产代际和比例不统一；Home interior 仍是功能原型；自动化 gate 偏工程可用而非艺术方向批准。
- V02.37 的 Round153 第二批 production 资产保留为资产库存，但不得进入 runtime 扩张或 final approval；`V02-STORYBATCH-004/005` 直到 V02.39 完成 runtime visual match 和 PM / Art Direction gate 前保持阻塞。
- 新视觉状态语义固定为：`production` 表示资源可集成；`functional_pass` 表示能玩且入口通；`visual_scaffold` 表示工程层级存在但未达目标画面；`art_target_locked` 表示 1280 目标画面和 Godot 可执行视觉合同获批；`runtime_visual_match` 表示运行时截图足够贴近目标画面；`final_approved` 必须有同轮 1280 runtime proof、真实入口、artpass003 / target frame 贴近度和 PM / Art Direction 判定。
- A-Z 记忆宫殿继续保留为世界结构，但在首屏视觉中必须降为生活物件和环境线索；完整 26 锚点可以作为探索层存在，不得在第一屏以系统索引、裸字母牌、课程入口或打卡目标压过小镇居住感。

## 下一阶段目标

V02.39 的目标不是修补当前截图，也不是继续批量接 story prop，而是重建 Godot 视觉生产体系：先形成可人工批准的 1280x720 target gameplay frame，再把它拆成 Godot 可执行的 visual layout / layer / perspective / density 合同，随后让 `TownStage` 运行时截图匹配该目标，最后再恢复真实入口和内容扩张。

新的生产顺序固定为：`target 1280 frame -> visual layout contract -> unified asset kit -> runtime static-frame match -> gameplay binding -> art-direction gate`。旧顺序 `world map data -> cell/place/anchor/NPC 全量摆放 -> 叠加美术 -> 截图 gate` 暂停作为 final visual approval 路线。后续仍不得重新规划 A-Z、把 School 做成课程空间、扩远郊为 P0、让地图变成可见瓦块，或把重构作为功能倒退的理由。

## 持续门槛

- 每个可玩功能必须有玩家可见入口，隐藏 contract 按钮不能作为完成依据。
- 每个素材必须有稳定逻辑 asset ID，不能由玩法脚本硬编码具体路径。
- 每个新词必须绑定 `letter`、`core_anchor_id`、`world_place_id`、`story_memory`、`visual_hook` 和 `review_path`。
- 每次 UI 或美术大改必须补截图验收，不能只跑 headless。
- 儿童端不得出现失败惩罚、课程压力、家长报告口吻、开放陌生人社交或工程调试文案。
- 隐藏网格不得在孩子端变成可见瓦块、坐标、调试路径或编辑器术语；scene 组件化不得破坏已完成的可见入口、保存结构和 A-Z 记忆宫殿稳定编码。
