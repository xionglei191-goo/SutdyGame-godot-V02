以下原计划保留用于追溯早期技术架构、数据结构和已实现 stub。凡与当前路线冲突时，以 `todo.md` 和 `docs/12_V02开发路线.md` 为准。

> 项目：StudyGame V02
> 目标：把 V02 从产品框架推进到可执行开发计划，定义数据结构、系统接口、任务清单、文档要求、测试标准和远景开发方向。
> 范围：计划覆盖到第一阶段主循环可跑通，不在本阶段要求真实联网、真实 AI、商业化或完整内容库。

## 1. 阶段边界

V02 第一阶段的结束状态：

- 可以在 Godot 编辑器中编辑世界地图，并导出运行时 JSON。
- 运行时能从 JSON 生成世界地图、地点、道路、碰撞、交互点和 A-Z anchor。
- Home / Pet / Shop / Coins / Memory Card / Letter Snake 的最小主循环可跑通。
- Voice 和 AI 只使用本地 stub 跑通接口，不接真实 TTS、录音权限、跟读或大模型。
- 家长后台能读取本地学习摘要 stub，但不进入孩子主流程。
- 远景能力已有接口或数据预留，但不扩大第一阶段实现边界。

第一阶段不做：

- 真实账号。
- 云存档。
- 联网互访。
- 真实 AI NPC。
- 真实语音跟读评分。
- 商业化、内购、订阅。
- 多套美术主题。
- 完整教材内容库。

## 2. 数据架构总则

V02 采用混合数据模式：

- Godot `Resource` 负责编辑体验、Inspector 友好配置和 editor tool 操作。
- JSON 负责运行时读取、自动测试、内容包、AI 管线和未来后台同步。
- Editor tool 负责 Resource / EditorOnly 节点 / JSON 的同步。

数据原则：

- 所有核心对象必须有稳定 `id`。
- 运行时不依赖 EditorOnly 节点。
- JSON 是运行时事实来源。
- Resource 是编辑体验来源。
- 每个 schema 都要声明第一阶段是否实现，以及未来扩展方向。

## 4. 核心服务接口

### 4.1 `AssetResolver`

职责：根据 `theme_id` 和逻辑 id 返回资源路径。

第一阶段要求：

- 返回 NPC、pet、tile、card、UI skin 占位资源。
- 禁止 gameplay 脚本直接硬编码具体资产路径。

### 4.2 `SaveService`

职责：本地保存 profile、GameState、LearningRecord。

第一阶段要求：

- Local 实现。
- 预留未来 Cloud 实现。

### 4.3 `MapEditorSyncService`

职责：同步 EditorOnly 节点、Resource 和 JSON。

第一阶段要求：

- 从 JSON 导入编辑节点。
- 从编辑节点导出 JSON。
- 支持保存前校验。

### 4.4 `MapValidationService`

职责：校验地图数据。

第一阶段要求：

- place 在合法 district。
- interaction cell 不在 occupied cell。
- shop / transport 连接道路。
- A-Z 完整且 route_order 唯一。
- quest focus place 存在。

### 4.5 `RuntimeMapBuilder`

职责：从 JSON 生成运行时地图。

第一阶段要求：

- 生成 TileMapLayer。
- 生成 landmark sprites。
- 生成 collision。
- 生成 hotspot。
- 生成 A-Z anchor markers。

### 4.6 `QuestEventService`

职责：启动、完成和记录事件。

第一阶段要求：

- 支持宠物需求事件。
- 支持商店购买事件。
- 支持小游戏入口事件。

### 4.7 `MemoryCardService`

职责：管理卡片状态。

第一阶段要求：

- Seen / Heard / Played / Collected / Shiny 状态可写入。
- 小游戏结果能推进卡片状态。

### 4.8 `PetService`

职责：管理宠物状态和需求。

第一阶段要求：

- 饥饿状态。
- 食物消耗。
- 开心反馈。
- 与 Shop/Quest/MemoryCard 奖励链连接。

### 4.9 `ShopService`

职责：购买食物、玩具、装饰。

第一阶段要求：

- coins 支付。
- 食物购买。
- 购买结果写入本地状态。

### 4.10 `MinigameService`

职责：启动小游戏、接收结果、发放奖励。

第一阶段要求：

- 启动 Letter Snake。
- 接收 score 和 target word 结果。
- 转换成 coins 和 card progress。

### 4.11 `VoiceService`

职责：TTS、录音、跟读预留。

第一阶段要求：

- mock TTS。
- mock recording。
- 不申请真实移动端权限。

### 4.12 `LLMClient`

职责：通用大模型接口。

第一阶段要求：

- stub 回复。
- 不接真实模型。
- 保留 prompt input/output 类型。

### 4.13 `NPCMemoryStore`

职责：读写 NPC 记忆文件。

第一阶段要求：

- 本地 stub。
- 可记录最近事件摘要。

### 4.14 `ParentDashboardStore`

职责：提供家长后台摘要数据。

第一阶段要求：

- 从 LearningRecord 和 NPC summary stub 读取本地摘要。

## 8. 测试与验收标准

第一阶段必须建立以下测试：

- schema load tests。
- JSON round-trip tests。
- map validation tests。
- runtime map smoke tests。
- A-Z uniqueness tests。
- Memory Card state tests。
- Pet/Shop/Coins loop tests。
- Minigame reward tests。
- Voice/AI stub tests。
- Parent dashboard visibility tests。

验收总标准：

- 孩子端没有强考核、课程表、家长报告。
- A-Z 核心编码不被推翻。
- 运行时数据来自 JSON。
- 编辑器工具能写回 JSON。
- 主循环可跑通。
- 远景能力已预留但不扩大首期实现。
